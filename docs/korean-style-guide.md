# 한국어 문서 문체 가이드

이 저장소의 한국어 문서는 사람이 직접 쓴 듯 읽혀야 한다. AI가 생성한 글에서 흔히
나타나는 군더더기 — 과한 볼드, 화살표 장식, 기계적인 단계 표기, 번역투 — 를 걷어내기
위한 기준을 정리한다. 새 문서를 쓰거나 기존 문서를 고칠 때 이 기준을 적용한다.

`/humanize <파일>` 슬래시 커맨드가 이 가이드를 그대로 따른다.

## 절대 건드리지 않는 것

문체를 다듬되 내용은 한 글자도 바꾸지 않는다. 다음은 모두 그대로 둔다.

- 코드 블록, 명령어, 옵션 플래그(`--generate-only`, `--firmware` 등)
- 파일 경로, 파일명 패턴(`{Project}_{Version}_bom.json` 등), URL
- CycloneDX·SPDX 필드명, 라이선스 ID, `purl` 같은 식별자
- 수치, 버전, 검출률 등 측정값
- 고유명사: SK텔레콤, cdxgen, syft, Trivy, Docker 등
- Mermaid 다이어그램 코드

라이선스 한 글자, 옵션 이름 하나가 틀리면 도구가 오작동한다. 의심스러우면 원문을 둔다.

## 걷어낼 패턴

우선순위가 높은 순서다(우리 문서에서 자주 나타나는 순).

### 1. 시각 장식 남용

가장 흔한 문제다.

- **볼드 과다** — 한 문단에 볼드가 너덧 개씩 박혀 있으면 강조의 의미가 사라진다.
  정말 핵심인 한두 군데만 남기고 푼다.
  - 고치기 전: `**프로젝트 이름**과 **버전**을 입력하고 **스캔 대상**을 골라 **스캔 실행**을 누른다`
  - 고친 뒤: `프로젝트 이름과 버전을 입력하고, 스캔 대상을 골라 실행을 누른다`
- **화살표(→) 장식** — "실행 → 스캔 → 다운로드"처럼 흐름을 화살표로 잇는 표현은
  문장으로 푼다. 다이어그램 안에서 쓰는 화살표는 그대로 둔다.
- **가운뎃점(·) 나열 남발** — "고지문·SBOM·보안 보고서·위험 보고서"처럼 길게 잇지
  말고, 자연스러운 문장이나 목록으로 바꾼다.

### 2. 구조적 AI 패턴

- **기계적 단계 표기** — "1단계 — …", "2단계 — …" 식의 라벨과 그 안의 1·2·3 불릿이
  겹치면 과하다. 짧은 안내는 문단으로 풀고, 정말 순서가 중요한 절차만 번호를 남긴다.
- **불필요한 목차** — 화면 한두 개 분량의 짧은 문서에 자동 생성형 목차를 달지 않는다.
- **표 남용** — 두세 줄짜리 정보까지 표로 만들지 않는다. 비교·참조표처럼 표가 정말
  나은 곳에만 쓴다.

### 3. 문장 리듬

- 모든 문장이 비슷한 길이로 끝나면 기계가 쓴 티가 난다. 긴 문장과 짧은 문장을 섞는다.
- "~합니다"가 연속으로 이어지면 종결을 바꿔본다.

### 4. 번역투

우리 문서엔 많지 않지만, 눈에 띄면 고친다.

- "~를 통해" → "~로", "~으로"
- "~에 대해 논의할 필요가 있다" → "~를 논의해야 한다"
- "~에 의해 생성된" → "~가 만든"
- 이중 피동("~되어지다") → 능동 또는 단일 피동

## 변경 강도

문체만 손질하고 의미·구조·정보는 100% 보존한다. 한 문서를 고친 분량이 지나치게 많으면
(im-not-ai의 30% 경고 기준 참고) 내용을 바꾸고 있는 건 아닌지 다시 본다. 윤문이 아니라
재작성이 되면 멈춘다.
